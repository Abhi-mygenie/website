Zapier Offline Conversions — CR-15
Demo Scheduled (offline) · Demo Given · Purchase (offline) → Google Ads OCI + Meta CAPI
You ~2–3h build · do after go-live P0
▼
What this step does
When a lead does NOT self-book Calendly, or when they attend the demo, or when they pay offline (bank/UPI/cheque) — Zapier fires conversion signals directly to Google Ads (OCI) and Meta (CAPI). These are signals the website GTM tags can never see. This step builds those 3 Zaps.
Pre-requisites — complete before building any Zap
  • Freshsales field audit — identify 4 custom field names
    Open any Freshsales contact created from the new website → check which custom fields are populated. You need the internal field names for:
    • gclid — Google Click ID (e.g. cf_gclid)
    • fbclid — Facebook Click ID (e.g. cf_fbclid)
    • fbp — Facebook browser cookie (e.g. cf_fbp)
    • event_id — lead UUID from MongoDB (e.g. cf_lead_id) — most important
    Share these names with Dev to confirm they match what the backend sends.
  • Create "Demo given" conversion action in Google Ads
    Google Ads → Goals → Conversions → New conversion action → Import from clicks (offline)
    Name: Demo given · Category: Book appointment · Value: ₹2,000 · Count: One per click
    Add to goal: "Book appointment" (same goal as Demo Scheduled).
    This action does not exist yet — must be created before Zap 2 can fire.
  • Confirm Calendly calendar names for Zap 1 filter
    Zap 1 needs to distinguish between bookings made via the website widget (already tracked by demo_booked) and bookings made by the sales team on behalf of the user.
    Confirm: what is the name/URL of the sales team's Calendly calendar vs the embedded website calendar?
  • Confirm Freshsales deal stage names for Zap 2 + Zap 3
    • Zap 2 trigger: what is the exact deal stage name for "demo was given / attended"? (e.g. Demo Given, Demo Attended)
    • Zap 3 trigger: what is the exact deal stage name for "customer paid offline"? (e.g. Won, Closed Won)
    These must be confirmed before building the Zap triggers.
Zap 1 — Demo Scheduled (offline path only)
When it fires
Sales team manually books Calendly on behalf of a user who did NOT self-book on the website. The website's demo_booked tag did NOT fire for this lead. No deduplication risk — mutually exclusive with Zap.
  • Zap 1 trigger: Calendly → Invitee Created (filter: sales team calendar only)
    Zapier → New Zap → Trigger: Calendly → Invitee Created → filter to fire only for the sales team calendar (not the website-embedded one). Use the calendar name confirmed in pre-requisites.
  • Zap 1 → Action A: Google Ads OCI → "Book appointments"
    Google Ads → Create Click Conversion action:
    • conversion_name: Book appointments
    • gclid: [Freshsales contact cf_gclid field]
    • conversion_time: Calendly invitee.created_at in IST (ISO 8601: 2026-06-22T14:30:00+05:30)
    • conversion_value: 1000
    • order_id: [lead UUID from Freshsales cf_lead_id] — prevents duplicate counting
  • Zap 1 → Action B: Meta CAPI → "Schedule" event
    Meta Conversions API action:
    • event_name: Schedule
    • event_id: [lead UUID from Freshsales cf_lead_id] — dedup key
    • event_time: Unix timestamp of booking
    • event_source_url: https://www.mygenie.online/
    • user_data.ph: contact phone · user_data.em: contact email · user_data.external_id: phone with +91
    • custom_data.value: 1000 · custom_data.currency: INR
Zap 2 — Demo Given (exclusively offline)
When it fires
Customer actually attended the demo (not just booked it). Freshsales deal stage changes to "Demo Given" (confirm exact name). This is a higher-intent signal than booking — recommended value ₹2,000.
  • Zap 2 trigger: Freshsales → Deal stage changed to "Demo Given"
    Zapier → New Zap → Trigger: Freshsales → Deal Updated → filter: stage = [confirmed stage name from pre-requisites].
  • Zap 2 → Action A: Google Ads OCI → "Demo given" (new conversion action)
    • conversion_name: Demo given (the new action created in pre-requisites)
    • gclid: [Freshsales cf_gclid]
    • conversion_time: demo appointment time, IST
    • conversion_value: 2000
    • order_id: [lead UUID]
  • Zap 2 → Action B: Meta CAPI → "ViewContent" (or custom "Demo Given")
    • event_name: ViewContent (or custom: Demo Given)
    • event_id: [lead UUID] + _demo_given — append suffix so Meta doesn't deduplicate against the Schedule event for same lead
    • user_data: phone, email, external_id, fbc, fbp
    • custom_data.value: 2000 · custom_data.currency: INR
Zap 3 — Purchase (offline payment path)
When it fires
Customer pays via bank transfer, UPI, or cheque (not through Razorpay on website). Freshsales deal marked "Won". Deduplication required — same lead could theoretically appear in both online and offline paths. Use order_id = lead_uuid + "_purchase" to prevent double-counting.
  • Zap 3 trigger: Freshsales → Deal stage changed to "Won" (filter: offline payment method)
    Zapier → New Zap → Trigger: Freshsales → Deal Updated → filter: stage = Won AND payment_method ≠ Razorpay (or filter by specific offline indicator in Freshsales).
  • Zap 3 → Action A: Google Ads OCI → "Purchased"
    • conversion_name: Purchased (existing conversion action — 19 historical conversions)
    • gclid: [Freshsales cf_gclid]
    • conversion_time: payment date, IST
    • conversion_value: actual deal value from Freshsales (NOT fixed — use variable value for value-based bidding)
    • order_id: [lead UUID] + _purchase — prevents dedup with online Razorpay purchase
  • Zap 3 → Action B: Meta CAPI → "Purchase" event
    • event_name: Purchase
    • event_id: [lead UUID] + _purchase
    • user_data: phone, email, external_id, fbc, fbp
    • custom_data.value: actual deal value · custom_data.currency: INR
